# 工作流规则

## 项目目标

`easy-specflow-zh` 的目标是为 AI 编码客户端提供一套统一、简单、中文优先的中长任务推进规则。

这套规则强调：

1. 平台中立
2. 文件驱动
3. 状态驱动
4. 统一目录约定
5. 适合跨 agent 协作

## 五阶段定义

| 阶段 | 目的 | 核心产物 |
| --- | --- | --- |
| `spec` | 澄清目标、范围、限制和验收标准 | `任务说明.md` |
| `plan` | 拆解步骤、明确风险、约定验证方式 | `执行计划.md` |
| `execute` | 按计划推进实现并持续记录 | `进度记录.md` |
| `verify` | 检查实现是否满足要求 | `进度记录.md` 中的验证部分 |
| `accept` | 形成最终验收结论和收尾记录 | `验收记录.md` |

## 每阶段产出要求

### `spec`

至少写清：

1. 任务背景
2. 目标
3. 范围与非目标
4. 约束与边界
5. 验收标准

### `plan`

至少写清：

1. 步骤列表
2. 每步目标
3. 影响范围
4. 验证方式
5. 风险与回滚思路

### `execute`

至少记录：

1. 本次执行目标
2. 实际改动摘要
3. 验证动作
4. 验证结果
5. 风险、阻塞和下一步

### `verify`

至少说明：

1. 对照了哪些验收标准
2. 还有哪些未通过项
3. 是否需要回退到 `execute`

### `accept`

至少写清：

1. 最终交付内容
2. 验收结论
3. 遗留问题
4. 后续建议

## 协作建议

1. 每次只推进一个明确阶段或一个明确步骤。
2. 先更新共享文档，再依赖聊天上下文。
3. 多 agent 接力时，优先更新 `state.json.lastAction`。
4. 阶段切换前先确认对应产物是否已经补齐。

## 过程护栏

1. 不得把五阶段文档一次性补满后直接标记完成，除非用户明确要求只生成初稿。
2. `执行计划.md` 是执行控制面，不是装饰文档；步骤完成后必须同步勾选。
3. `verify` 的作用是判断是否满足验收标准，而不是补一个形式上的结论。
4. `accept` 的作用是收口，不是跳过未完成步骤或未解决阻塞。
5. `status=done` 只表示任务真正完成收口，不表示“代码大致完成”或“有条件通过”。
6. 如果存在环境阻塞、待补验证或未完成计划步骤，任务必须保持 `status=active`。

## 跨 agent 使用原则

1. 任何 agent 都应读取同一套规则、模板、提示词和 `.specflow/`。
2. 任何 agent 都不应创建私有副本或平行目录。
3. 如果某个 agent 做了局部工作，也要把结果写回共享文档。
4. 若文档、状态和实现不一致，以最新校准后的共享文件为准。
