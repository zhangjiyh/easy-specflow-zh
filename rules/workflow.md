# 工作流规则

## 项目目标

`easy-specflow-zh` 的目标，是为 AI 编码客户端和人工协作场景提供一套统一、简单、中文优先的长任务执行协议。

这套协议强调：

1. 平台中立
2. 文件驱动
3. 状态驱动
4. 阶段护栏
5. 验证证据化
6. 适合跨 agent 协作

## 六阶段定义

| 阶段 | 中文说明 | 核心产物 |
| --- | --- | --- |
| `spec` | 需求定义：澄清目标、范围、限制和验收标准 | `任务说明.md` |
| `repo-map` | 代码库定位：确认代码边界、入口文件、直接相关文件与潜在影响面 | `代码库地图.md` |
| `plan` | 执行计划：拆解步骤、明确修改范围、约定验证方式与完成标准 | `执行计划.md` |
| `execute` | 执行实现：按计划推进实现并持续记录执行结果 | `进度记录.md` |
| `verify` | 验证汇总：汇总验证证据并判断是否满足需求与计划 | `验证记录.md` |
| `accept` | 验收结论：基于前序证据形成最终验收结论和收尾记录 | `验收记录.md` |

## 每阶段产出要求

### `spec`

至少写清：

1. 任务背景
2. 目标
3. 范围与非目标
4. 约束与边界
5. 验收标准

### `repo-map`

至少写清：

1. 关键目录
2. 关键入口文件
3. 直接相关文件
4. 潜在影响文件
5. 搜索关键词或路径提示
6. 风险点与不确定项

### `plan`

至少写清：

1. 步骤列表
2. 每步目标
3. 修改范围 / 涉及文件
4. 执行动作
5. 验证方式
6. 完成标准
7. 当前状态

### `execute`

至少记录：

1. 对应步骤编号
2. 本次执行目标
3. 实际改动摘要
4. 影响文件
5. 验证命令
6. 验证结果
7. 问题、阻塞和下一步

### `verify`

至少说明：

1. 验证范围
2. 验证命令
3. 预期结果
4. 实际结果
5. 是否通过
6. 失败摘要
7. 遗留风险或待补验证项

### `accept`

至少写清：

1. 任务目标回顾
2. 验收依据
3. 验收结论
4. 未解决问题
5. 已知风险
6. 后续建议

## 协作建议

1. 每次只推进一个明确阶段或一个明确步骤。
2. 先更新共享文档，再依赖聊天上下文。
3. 多 agent 接力时，优先更新 `state.json.lastAction`。
4. 阶段切换前先确认对应产物是否已经补齐。
5. 交接时明确当前步骤、验证状态和待补项，避免任务状态漂移。

## 过程护栏

1. 不得把六阶段文档一次性补满后直接标记完成，除非用户明确要求只生成初稿。
2. `repo-map` 是计划前置输入，不是可有可无的附录。
3. `执行计划.md` 是执行控制面，不是装饰文档；步骤完成后必须同步更新状态。
4. `verify` 的作用是形成证据，而不是补一个形式上的“已验证”结论。
5. `accept` 的作用是基于证据收口，不是跳过未完成步骤、未关闭失败项或未解决阻塞。
6. `status=done` 只表示任务真正完成收口，不表示“代码大致完成”或“有条件通过”。
7. 如果存在环境阻塞、待补验证或未完成计划步骤，任务必须保持 `status=active`。

## 跨 agent 使用原则

1. 任何 agent 都应读取同一套规则、模板、提示词和 `.specflow/`。
2. 任何 agent 都不应创建私有副本或平行目录。
3. 如果某个 agent 做了局部工作，也要把结果写回共享文档。
4. 若文档、状态和实现不一致，以最新校准后的共享文件为准。
