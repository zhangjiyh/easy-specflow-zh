# 使用流程说明

这份说明描述一个典型的使用周期，帮助团队把任务稳定推进到 `.specflow/` 中。

## 1. 初始化 `.specflow/`

在目标项目根目录执行：

```bash
bash scripts/init-workflow.sh
```

如果脚本仓库和目标项目分离，请先切换到目标项目根目录，再调用脚本绝对路径。

初始化后会得到：

```text
.specflow/
├─ index.json
├─ archive/
├─ trash/
└─ tasks/
```

## 2. 新建任务

```bash
bash scripts/specflow.sh new "登录与鉴权重构"
```

系统会创建任务目录，并在后续阶段逐步补齐协议产物：

```text
.specflow/tasks/TYYYYMMDD-001-登录与鉴权重构/
├─ 任务说明.md
├─ 代码库地图.md
├─ 执行计划.md
├─ 进度记录.md
├─ 验证记录.md
├─ 验收记录.md
└─ state.json
```

## 3. 编写 `spec`（需求定义）

首先完成 `任务说明.md`，至少写清：

1. 任务背景和目标。
2. 范围与非目标。
3. 约束和禁改项。
4. 验收标准。

完成后，可以把 `state.json.stage` 保持在 `spec`，直到关键信息足够进入代码库定位阶段。

## 4. 编写 `repo-map`（代码库定位）

在写计划前，先补全 `代码库地图.md`：

1. 标出关键目录和入口文件。
2. 识别直接相关文件和潜在影响文件。
3. 记录搜索关键词、路径提示和风险点。

这个阶段的目标不是写大而全的设计文档，而是让计划建立在真实代码上下文之上。

## 5. 制定 `plan`（执行计划）

当目标和代码边界都清楚后，补全 `执行计划.md`：

1. 拆出阶段内步骤。
2. 标明每一步的目标、修改范围、执行动作、验证方式和完成标准。
3. 使用统一的步骤勾选格式，便于 `next-step.sh` 给出建议。

计划确认后，把 `state.json.stage` 更新到 `plan`。

## 6. 执行 `execute`（执行实现）

进入执行阶段后：

1. 默认连续处理剩余明确步骤，直到全部计划步骤完成或遇到阻塞。
2. 把实际改动、影响文件、验证命令、验证结果、问题和阻塞写入 `进度记录.md`。
3. 当计划中某一步完成时，记得同步更新 `执行计划.md` 中的当前状态。
4. 默认在整轮执行结束后统一总结，而不是每完成一步就单独停下来汇报。

执行中可随时查看：

```bash
bash scripts/status.sh
bash scripts/next-step.sh
```

## 7. 汇总 `verify`（验证汇总）

所有关键步骤完成后，进入 `verify`：

1. 对照 `任务说明.md` 的验收标准逐项检查。
2. 对照 `执行计划.md` 检查是否存在未完成项。
3. 对照 `进度记录.md` 汇总验证命令、预期结果、实际结果、是否通过、失败摘要和待补验证项。
4. 把结构化结论写入 `验证记录.md`。

基础校验可用：

```bash
bash scripts/verify.sh
```

## 8. 完成 `accept`（验收结论）

验证通过后，整理 `验收记录.md`：

1. 回顾任务目标。
2. 列出验收依据。
3. 给出验收结论。
4. 说明遗留问题、已知风险和后续建议。

只有在结论为“通过”且没有待补验证项时，才应该把 `state.json.status` 更新为 `done`。

## 9. 归档或恢复

常用命令：

```bash
bash scripts/specflow.sh archive
bash scripts/specflow.sh restore T20260329-001
bash scripts/specflow.sh delete
```

建议：

1. 已完成且暂不继续推进的任务归档到 `archive/`。
2. 暂时废弃但可能恢复的任务移到 `trash/`。
3. 所有归档和恢复动作都同步更新 `index.json` 和 `state.json`。
