# 跨 agent 协作说明

这份说明展示多个 agent 或人工如何共用同一套 `.specflow/` 目录推进任务。

## 协作原则

1. 同一个任务只维护一个目录。
2. 所有 agent 共用同一个 `state.json`。
3. 阶段切换以 `state.json.stage` 为准。
4. `任务说明.md`、`代码库地图.md`、`执行计划.md`、`进度记录.md`、`验证记录.md`、`验收记录.md` 都是共享事实来源。
5. 不为不同 agent 建立重复目录或重复模板。

## 哪些文件是共享事实来源

1. `state.json`
   用来记录当前阶段、当前状态和最近动作摘要。
2. `任务说明.md`
   用来记录目标、范围、约束和验收标准。
3. `代码库地图.md`
   用来共享代码边界、入口文件和潜在影响面共识。
4. `执行计划.md`
   用来约束步骤、修改范围、验证方式和完成标准。
5. `进度记录.md`
   用来记录每一步真实执行结果、影响文件和即时验证结果。
6. `验证记录.md`
   用来汇总验证命令、预期结果、实际结果、失败摘要和待补验证项。
7. `验收记录.md`
   用来基于前序证据给出最终结论。

## 哪些字段必须更新

交接前至少同步以下信息：

1. `state.json.stage`
2. `state.json.updatedAt`
3. `state.json.lastAction.agent`
4. `state.json.lastAction.summary`
5. 当前步骤在 `执行计划.md` 中的状态
6. 本次执行结果在 `进度记录.md` 中的记录
7. 本次验证结果在 `验证记录.md` 中的状态

## 如何避免状态漂移

1. 不把阶段判断留在聊天里，要写回 `state.json`。
2. 不把文件边界共识留在口头里，要写回 `代码库地图.md`。
3. 不把“做过验证”留成口头结论，要写回 `验证记录.md`。
4. 如果文档、状态和实现不一致，先修正文档与 `state.json`，再继续执行。

## 示例流程

### 场景

任务标题：`登录与鉴权重构`

任务目录：

```text
.specflow/tasks/T20260329-001-登录与鉴权重构/
```

### 第一步：GLM / Claude Code 先写 `spec` 与 `repo-map`（需求定义与代码库定位）

可以先完成：

1. 在 `任务说明.md` 中写清目标、范围、限制和验收标准。
2. 在 `代码库地图.md` 中识别后端鉴权入口、前端登录页、请求封装和潜在影响文件。
3. 把 `state.json.stage` 更新为 `repo-map` 或 `plan` 前的真实阶段。

### 第二步：Codex 细化 `plan`（执行计划）

Codex 接手后：

1. 阅读 `任务说明.md` 与 `代码库地图.md`。
2. 在 `执行计划.md` 中拆出后端登录接口、鉴权拦截器、前端接入、联调验证等步骤。
3. 为每一步补全验证方式和完成标准。
4. 在 `state.json.lastAction` 中写明这次计划细化的摘要。

### 第三步：OpenCode 执行 `execute`（执行实现）

OpenCode 根据计划连续执行剩余步骤，直到全部计划步骤完成或遇到阻塞：

1. 修改代码。
2. 在 `进度记录.md` 中写入对应步骤编号、改动摘要、影响文件、验证命令与结果。
3. 每完成一步就更新 `执行计划.md` 中对应步骤的勾选项与当前状态。
4. 把 `state.json.stage` 保持为 `execute`。

### 第四步：Codex 汇总 `verify`（验证汇总）

Codex 再次接手时：

1. 对照 `任务说明.md` 的验收标准检查实现。
2. 对照 `代码库地图.md` 检查修改是否越界。
3. 对照 `执行计划.md` 和 `进度记录.md` 汇总验证命令、预期结果、实际结果、失败摘要和待补验证项。
4. 把验证结论写回 `验证记录.md`，并将 `state.json.stage` 更新为 `verify`。

### 第五步：人工或任一 agent 完成 `accept`（验收结论）

最后一步：

1. 在 `验收记录.md` 中写明验收依据、最终结论、遗留问题和已知风险。
2. 如果结论为“通过”且没有待补验证项，可将 `state.json.status` 更新为 `done`。
3. 如果结论为“有条件通过”，应保持 `status=active`，不能等同于完全完成。

## 一个简单交接清单

交接前快速确认：

1. 当前阶段是否写回 `state.json.stage`
2. 当前步骤是否写回 `执行计划.md`
3. 本次修改是否写回 `进度记录.md`
4. 本次验证是否写回 `验证记录.md`
5. 是否仍有阻塞或待补验证项

## 这个协作模型为什么简单

因为它只依赖三样东西：

1. 同一个任务目录。
2. 同一份 `state.json`。
3. 同一套规则、模板、提示词和证据文档。

这让不同 agent 的接力成本很低，也更容易做审阅、交接和回溯。
