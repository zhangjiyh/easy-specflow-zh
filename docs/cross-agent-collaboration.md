# 跨 agent 协作说明

这份说明展示多个 agent 如何共用同一套 `.specflow/` 目录推进任务。

## 协作原则

1. 同一个任务只维护一个目录。
2. 所有 agent 共用同一个 `state.json`。
3. 阶段切换以 `state.json.stage` 为准。
4. 任务说明、执行计划、进度记录、验收记录都是共享产物。
5. 不为不同 agent 建立重复目录或重复模板。

## 示例流程

### 场景

任务标题：`登录与鉴权重构`

任务目录：

```text
.specflow/tasks/T20260329-001-登录与鉴权重构/
```

### 第一步：Claude Code 先写任务说明与计划

Claude Code 可以先完成：

1. 在 `任务说明.md` 中写清目标、范围、限制和验收标准。
2. 在 `执行计划.md` 中拆出后端鉴权、前端接入、联调验证等步骤。
3. 把 `state.json` 更新为：

```json
{
  "stage": "plan",
  "status": "active"
}
```

### 第二步：Codex 检查计划与实现约束

Codex 接手后：

1. 阅读 `任务说明.md` 与 `执行计划.md`。
2. 校对边界、风险和回滚策略。
3. 如有必要，细化某一步的验证方法。
4. 在 `state.json.lastAction` 中写明这次检查的摘要。

### 第三步：OpenCode 执行部分代码改动

OpenCode 根据计划执行某一个步骤：

1. 修改代码。
2. 在 `进度记录.md` 中写入改动摘要和验证摘要。
3. 勾选 `执行计划.md` 中对应的步骤。
4. 把 `state.json.stage` 更新为 `execute`。

### 第四步：Codex 做 verify

Codex 再次接手时：

1. 对照 `任务说明.md` 的验收标准检查实现。
2. 对照 `执行计划.md` 检查是否仍有未完成项。
3. 把验证结论写回 `进度记录.md` 的验证部分。
4. 把 `state.json.stage` 更新为 `verify`。

### 第五步：最终整理验收记录

任一 agent 都可以完成最后一步：

1. 在 `验收记录.md` 中写明交付结果、遗留问题和验收结论。
2. 将 `state.json.status` 更新为 `done`。
3. 若任务结束且暂不再修改，可再执行归档脚本。

## 这个协作模型为什么简单

因为它只依赖三样东西：

1. 同一个任务目录。
2. 同一份 `state.json`。
3. 同一套规则、模板和提示词。

这让不同 agent 的接力成本很低，也更容易做审阅、交接和回溯。
